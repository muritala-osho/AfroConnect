# AfroConnect

## Overview
AfroConnect is a dating and social application designed for the African diaspora. Its primary purpose is to connect individuals within this community. The project is structured into three independent components: a Node.js/Express backend API, an Expo/React Native mobile frontend, and a React-based admin dashboard. Key capabilities include real-time chat, personalized icebreakers, advanced user management and moderation, and robust analytics. The platform aims to foster meaningful connections and build a vibrant community for the African diaspora globally.

## System Architecture
AfroConnect is built as a monorepo containing three distinct applications:

**1. Backend (Node.js + Express):**
*   **Stack**: Node.js, Express, MongoDB (Mongoose for ORM), Socket.io for real-time communication.
*   **Purpose**: Provides RESTful APIs for mobile and admin clients, handles WebSocket connections for chat and calls, and manages data persistence.
*   **Core Features**:
    *   User authentication and authorization (JWT).
    *   Chat functionality with voice notes, media sharing, and reactions.
    *   Personalized icebreaker suggestions based on user interests.
    *   User profile management, including location, interests, and verification status.
    *   Admin API endpoints for user management, content moderation, reports, and analytics.
    *   Notification services (FCM for Android, APNs VoIP for iOS) for calls and general push notifications.
    *   Support ticket system with agent assignment.
    *   Live location updates and daily matching algorithms.
    *   In-app purchase (IAP) endpoints: `POST /subscription/validate-receipt` and `POST /subscription/restore-purchases` for Google Play / Apple IAP receipt validation. Server-side validation uses `GOOGLE_SERVICE_ACCOUNT_JSON` (Android) and `APPLE_IAP_SHARED_SECRET` (iOS); falls back to trusting client if credentials are absent. Both endpoints persist `premium.originalTransactionId` (iOS) and `premium.purchaseToken` (Android) so subscription lifecycle webhooks can locate the user.
    *   Subscription lifecycle webhooks: `POST /subscription/webhook/apple` accepts Apple App Store Server Notifications V2 (`{ signedPayload }` JWS, ES256 leaf-cert verified by default; disable with `APPLE_WEBHOOK_VERIFY_SIGNATURE=false`). `POST /subscription/webhook/google` accepts Google Play Real-Time Developer Notifications via Pub/Sub push (`{ message: { data: base64-json } }`); optionally protected by `GOOGLE_RTDN_TOKEN` query/header. Handlers map renew/cancel/expire/refund/grace events to `User.premium` state via `services/iapWebhookService.js`.
    *   Translation caching (in-memory, 24h TTL, 500-entry LRU) with graceful MYMEMORY WARNING detection returning user-friendly 429 responses.
    *   All chat socket emissions (`chat:new-message`) call `message.toJSON()` before emitting, ensuring all schema fields (including GIF fields) are properly serialized for all receivers.
    *   GIF messages persist `gifUrl`, `gifPreview`, `gifWidth`, `gifHeight`, `gifSource` in the Message schema and include them in all socket payloads and REST history responses.

**2. Frontend (Expo / React Native):**
*   **Stack**: Expo, React Native, TypeScript.
*   **UI/UX**: Features a flat conversation row style (Tinder/Bumble-like), gradient hero cards, and 2x2 benefits grids for specific screens. Uses colored icon bubbles for profile sections, matching the edit profile color scheme. Voice note UX includes preview, pause/resume, variable speed playback, continuous auto-play, and waveform scrubbing. Image viewer supports pinch-to-zoom, double-tap-zoom, and a swipeable gallery.
*   **Technical Implementations**:
    *   Optimistic UI updates for media sends and chat messages.
    *   Real-time chat: `handleNewMessage` uses `String()` comparison for matchId to handle ObjectId/string type differences. Uses `isNearBottomRef` to only auto-scroll when user is near the latest message, preventing scroll hijacking when reading history.
    *   Keyboard: on Android (edge-to-edge enabled), `KAVController` uses `keyboardVerticalOffset={insets.bottom}` to account for the navigation bar. `inputPaddingAnim` smoothly animates between `insets.bottom` (keyboard hidden) and `4` (keyboard visible).
    *   Integration with `react-native-callkeep` and `react-native-voip-push-notification` for native call UI (CallKit/ConnectionService) even when the app is killed.
    *   Firebase Messaging for background message handling and push notifications.
    *   Video verification flow with liveness detection (blink, head turns) using MediaPipe-style landmark gating and Lottie animations for guided prompts.
    *   Device timezone synchronization for accurate local time display.
    *   Robust error handling with tap-to-retry for failed media uploads.

**3. Admin Dashboard (React + Vite + Tailwind CSS):**
*   **Stack**: React 18, Vite, Tailwind CSS, Recharts, Lucide React, TypeScript.
*   **Purpose**: Provides administrative control over the platform.
*   **UI/UX**: Responsive design with a sidebar navigation. Features dark/light mode toggle.
*   **Features**:
    *   JWT Admin Authentication.
    *   Dashboard with live statistics and charts (Recharts).
    *   User Management: comprehensive table with search, filter, ban, suspend, delete, and verification controls.
    *   Reports & Moderation: content moderation queue for user-reported content (photos, stories, chat images) with approve/reject functionality.
    *   ID Verification: queue for reviewing and approving/rejecting user verification videos.
    *   Analytics: graphs for user growth, engagement, and activity.
    *   Broadcasts: send push notifications to segmented user groups.
    *   Support Desk: centralized system for managing support tickets, with agent assignment and chat interface.
    *   Payments & Revenue tracking.
    *   Premium Members panel: per-user subscription state from `GET /admin/premium-members` showing source (iOS/Android/web/admin), plan, expiry, days-remaining, auto-renew status, last webhook event, transaction identifiers, and active feature flags. Filters by source, plan, status (expiring soon / cancelled-but-active / expired), auto-renew, plus search by name/email/transaction ID/purchase token.
    *   Grant Free Premium: admins can comp users with the full Premium feature set via a header button on the Premium Members panel. AfroConnect ships **a single Premium tier** (no Plus/Gold/Platinum), so the modal only asks for duration + optional reason. Backed by `POST /admin/users/:userId/grant-premium` (`durationDays`, optional `reason` — features come from `iapWebhookService.PREMIUM_FEATURES`, `premium.plan` is set to `admin_grant`, `premium.source` to `admin`), `POST /admin/users/:userId/revoke-premium` (only revokes admin-granted premium — store-billed subscriptions must be cancelled at the store), and `GET /admin/users/lookup?q=` for typeahead search. Granting an active subscription extends the existing expiry. All grants/revokes write `category: SUBSCRIPTION` audit log entries with severity `high`.
    *   Expiring-grants tracking: the Premium Members summary cards include an **Admin Grants** count and a clickable **Grants Expiring** badge (shows admin-granted comps expiring within 7 days; clicking applies the `admin_expiring` status filter). A scheduled job (`runAdminGrantExpiryWarnings` in `backend/utils/scheduledJobs.js`, runs hourly) emails every `User { isAdmin: true }` once per grant when an admin comp is within 7 days of expiry, using `sendAdminGrantExpiryWarningEmail`. The notification flag (`premium.adminGrantExpiryWarningSentAt`) is reset whenever an admin re-grants/extends, so the next expiry triggers a fresh warning.

## Location & Permission Handling
*   **Permission cache**: `frontend/utils/locationPermission.ts` caches the OS foreground location permission in AsyncStorage with a 24h TTL (`getCachedPermissionStatus`, `requestAndCachePermission`, `clearPermissionCache`). Screens that need to know permission state (Discovery, Radar fetch) read from the cache instead of calling `Location.getForegroundPermissionsAsync()` on every focus, which removes the OS round-trip on hot mounts and makes the gate decision deterministic.
*   **Discovery gate**: when `/api/users/nearby` cannot find any usable origin (no request coords, no stored profile coords, no active passport, not in Global mode), it returns `{ users: [], requiresLocation: true }`. The mobile DiscoveryScreen renders a dedicated "Enable location to start discovering" empty state with an `Enable Location` primary button (uses `requestAndCachePermission`) and, for Premium users, a `Browse Globally` fallback. The previous behaviour silently returned an unfiltered list with no distance restriction — confirmed leak, now closed.
*   **New-user signup discovery bug**: `handleShareLocation` previously invoked `loadPotentialMatches()` directly after `updateProfile({ location })`; that captured the stale memoized closure (still no coords) and re-triggered the gate on the very first fetch. The fix sets `setRequiresLocation(false)`, resets `preferencesRef.current = ''`, and lets the dependent `useEffect` re-fire with the fresh `user.location.lat/lng`, which produces a re-memoized `loadPotentialMatches` that hits the backend with the new coordinates.

## Auth & Session Lifecycle
*   **Access token**: 24h JWT, signed with `{ id, tokenVersion, sessionId }` in `backend/routes/auth.js` `generateToken`. Stored client-side in `expo-secure-store` under key `auth_token`.
*   **Refresh token**: opaque random string, hash stored on the `Session` row (`refreshTokenHash`). Stored client-side under key `auth_refresh_token`. The Session has a sliding 30-day TTL (`expireAfterSeconds: 30*24*60*60` on `lastActive`), so any `/auth/refresh` call effectively renews session lifetime.
*   **Refresh-token rotation removed (re-confirmed 2026-04-29)** — rotation had been silently re-introduced with a 60s grace window and was again causing random sign-outs whenever `/auth/refresh` ran (the rotated token would fail to persist client-side on transient network errors, parallel page mounts in the admin dashboard, SecureStore write races on iOS, or app force-quits between request and storage write). The `/auth/refresh` route now ONLY bumps `session.lastActive` and issues a new access token — the refresh token itself stays the same for the lifetime of the Session (30-day TTL on `lastActive`). The route still accepts a `previousRefreshTokenHash` within its existing grace window so any sessions left over from when rotation was on can drain naturally without forcing those users to re-login. Sessions remain revocable via `/auth/logout` (deletes the Session row), via `tokenVersion` bumps on the User document (forces all sessions invalid), and via the Redis revocation set keyed by `revoked:${sessionId}` (checked in both `/auth/refresh` and `middleware/auth.js`). Multi-device login is unaffected — each device has its own Session row with its own refresh token. This matches the standard first-party mobile-app pattern used by Tinder, Bumble, and Hinge: short access tokens + long-lived non-rotating refresh tokens + server-side session revocation, where UX cost of rotation outweighs the marginal security benefit. Both clients (`frontend/utils/tokenManager.ts` and `admin-dashboard/services/adminApi.ts`) already handle a missing `refreshToken` in the response by leaving the stored one in place, so no client changes were needed.
*   **Client-side 401 handling**: `frontend/hooks/useApi.ts` (78–108) intercepts 401s on non-`/auth/*` endpoints, calls `tokenManager.refresh()`, and retries the original request once. `frontend/utils/tokenManager.ts` only triggers `onSessionExpiredCallback` (logout) when refresh returns 401/403 or `tokenRevoked: true`; transient network failures or 5xx responses keep tokens intact and let the next request retry.

## Android Build Plugins
*   **`frontend/plugins/withCallKeep.js`**: Injects the Android `ConnectionService` for `react-native-callkeep` (CallKeep needs a native Android service registered in the manifest to receive incoming VoIP calls).
*   **`frontend/plugins/withFirebaseManifestFix.js`**: Resolves an Android manifest merge conflict between `expo-notifications` and `@react-native-firebase/messaging`. ## Face Verification Gating
*   **`isFaceVerified` field** (`backend/models/User.js`): new Boolean field (default `false`, indexed) that is the dedicated gate flag for face verification. Set to `true` by the admin approve route and `false` by the reject route, always in sync with `verified` and `verificationStatus`.
*   **Backend gates** (enforced in all environments, not just production):
    *   `GET /users/nearby` re-fetches `isFaceVerified` and `verificationStatus` from MongoDB on every request. Any user where `isFaceVerified !== true || verificationStatus !== 'approved'` gets a `403 { verificationRequired: true, verificationStatus }` response. The gate uses a DB read instead of trusting the JWT payload to prevent token-craft bypass.
    *   `POST /match/swipe` applies the same check immediately before any swipe logic.
*   **Frontend gate** (`frontend/screens/DiscoveryScreen.tsx`): a `useFocusEffect` fires `GET /api/verification/status` and `fetchUser()` on every screen focus. If the result is not `approved`, the swipe deck is replaced by a full-screen gate UI:
    *   **Not verified / none**: shield icon, "Verify your face before seeing users", "Verify Now" CTA → navigates to VerificationScreen.
    *   **Pending**: amber clock icon, "Verification under review", "Refresh Status" button (re-polls the endpoint).
    *   **Rejected**: red X icon, "Verification failed", "Try Again" CTA → navigates to VerificationScreen.
*   **VerificationScreen banners** updated to mention that discovery and swiping are locked until approval.
*   **Type safety**: `isFaceVerified?: boolean` and `verificationStatus?` union type added to the `User` interface in `frontend/hooks/useAuth.tsx`.
*   **Per-user Notification History tab** (`admin-dashboard/views/UserNotificationLog.tsx` + `backend/models/NotificationLog.js` + `backend/utils/notificationLogger.js`) — every email and push notification sent to any user is captured in the `notificationlogs` collection with channel, type, subject/body, status (sent / failed / bounced / suppressed), Brevo `messageId` or Expo ticket id, and any error message. Coverage is automatic: `brevoSend` (in `emailService.js`) wraps every transactional email through the central helper and `sendExpoPushNotification` (in `pushNotifications.js`) wraps every push — neither requires call-site changes. When `userId` isn't passed explicitly, the logger resolves it via a 5-min in-memory cache keyed by recipient email (for email) or push token (for push). The admin user profile modal in `UserManagement.tsx` now has a **Notifications** tab fed by `GET /api/admin/users/:userId/notifications?page=&limit=&channel=email|push&status=`. The endpoint returns paginated items + per-channel sent/failed counts so support agents can verify "did we actually send it?" in one click — including the provider id for cross-referencing in Brevo or Expo's dashboards.

*   **Approve / Reject / Revoke notifications** — every admin action on a user's verification fires three things in parallel: (a) a transactional Brevo email using the templates in `backend/utils/emailService.js` (`sendVerificationApprovedEmail`, `sendVerificationRejectedEmail`, `sendVerificationRevokedEmail` — the revoke template was added when the gap was found), (b) an Expo push notification via `sendExpoPushNotification` gated on `user.pushToken && user.pushNotificationsEnabled !== false`, and (c) a `socket.io` emit on the user's room (`user:verified` / `user:verification-rejected`) so an open app updates its UI in real time. The admin dashboard hits `PUT /api/admin/verifications/:userId/approve|reject` (in `backend/routes/admin.js` ~lines 595-735) — these were initially silent and have been brought up to parity with the equivalent routes in `backend/routes/verification.js`. Revocation goes through `verificationService.revokeVerification` which now also calls `sendVerificationRevokedEmail` in addition to the existing in-app notification + push. All notification attempts are wrapped in try/catch and log on failure so a transient SMTP/push outage never aborts the database mutation.

## Discovery & Matching Behaviour
*   **Daily Super-Like cap** (`backend/routes/match.js` `/swipe` handler): 5/day for Premium, **1/day for Free**. Counter is `superlikecount:<userId>:<YYYY-MM-DD>` in Redis with TTL to UTC midnight; falls back to `user.superLikesDaily.{count, lastReset}` on the User doc when Redis is unavailable. Hitting the cap returns `403 { code: 'SUPERLIKE_LIMIT_REACHED', dailyCap, isPremium, message }` so the client can show an upsell. Free users were previously hard-blocked from super-likes; they now get exactly one per day to taste the feature without diluting Premium scarcity.
*   **Discovery / Radar parity** (`backend/routes/users.js` `/nearby`): the discovery-side `pendingIds` exclusion has been removed. Previously Discovery hid every user with any pending FriendRequest involving the current user, which silently blocked people who had already liked them — those users still appeared on the Love Radar but never on Discovery, producing the "I see them on Radar but not on Discovery" complaint. Discovery and Radar now share the same exclusion set: blocked, blocked-by, already-swiped (left or right), active matches, banned/suspended, incognito (Radar only). When a current-user right-swipe happens on a person who had a pending like to them, the existing `/swipe` handler turns it straight into a Match.
*   **Frontend coords normalization** (`frontend/screens/DiscoveryScreen.tsx` `loadPotentialMatches`): the `/users/nearby` request now reads coordinates from any of `user.location.lat/lng`, `user.location.coordinates.coordinates[0/1]` (GeoJSON Point), or `user.location.coordinates[0/1]` (legacy flat array). Before, only the flat `lat/lng` form was checked, so users whose profile only had the GeoJSON form sent the request without coordinates and silently relied on the backend's stored-coord fallback — fine for results, but it broke distance display on the cards and missed the discovery cache key. With normalization the URL is deterministic from the first call.
*   **Premium instant-load Discovery cache** (`frontend/screens/DiscoveryScreen.tsx`): for Premium users, the last batch of discovery cards (top 20) is persisted to AsyncStorage under `discovery_cache_v1:<userId>` with a 30-min TTL, scoped by `discoveryType` + `selectedCountry`. On the very first mount of the screen, the cache is hydrated synchronously so the deck renders immediately — no spinner — while `/users/nearby` fires in the background to refresh. Free users continue to see the loading state. This is part of the Premium value prop alongside Global, Passport, unlimited likes, etc.

## Chat Features
*   **Edit message** (text-only, 15-min window): Long-press own text bubble → context menu shows "Edit Message" only when `type === 'text'`, sender is self, message is not deleted, AND `Date.now() - createdAt <= 15 min`. Editor opens `EditMessageModal` (`frontend/components/chat/EditMessageModal.tsx`) which submits `PATCH /api/chat/message/:messageId` with `{ content }`. Backend (`backend/routes/chat.js` line 625) re-checks all the same rules server-side and emits `chat:message-edited` over socket.io with `{ messageId, matchId, content, edited, editedAt }`. The chat screen's socket listener and the local `setMessages` after the HTTP call both flip `edited: true` and persist `editedAt`. An italic "edited" label appears in the message footer for any message with `edited === true`.
*   **Per-message read receipts (three-state ticks)**: Server emits `chat:message-read` (and legacy `chat:read`) on the match room with `{ chatId, matchId, userId, readAt }` whenever the recipient marks messages as read (HTTP `PUT /api/chat/:matchId/read`, single-message `PUT /api/chat/message/:id/seen`, or socket `chat:read`). The chat screen flips own messages to `status: 'seen'` AND stamps `seenAt: readAt` (preferring an existing `seenAt` if one is already on the message, so we never overwrite the original read time). The bubble footer renders three states: single grey checkmark for `sent`, double white-ish checkmark for `delivered` (driven by the sender-room `chat:message-delivered` socket emit), and double blue checkmark (`#4FC3F7`) once it flips to `seen`.
*   **Lock-screen sender avatar (rich push notifications)**: Chat-message and missed-call pushes include `richContent: { image: <senderPhotoHttpsUrl> }` and `mutableContent: true` in the Expo push payload. `backend/utils/pushNotifications.js` `sendExpoPushNotification` whitelists `richContent`, `mutableContent`, and `categoryIdentifier`, sanitizing `richContent.image` to require an HTTPS URL (drops anything else to avoid silent FCM/APNs delivery failures). On Android (FCM v1, which is enabled via `useFcmV1: true`) Expo translates this to `notification.image` so the avatar renders as the big-picture / large-icon expansion. On iOS, `mutableContent: true` triggers the Notification Service Extension (NSE) hook so the image can be downloaded and attached on-device — **note: the current iOS build does not ship an NSE, so iOS quietly falls back to a plain notification (no error, no broken delivery). To enable iOS rich images, add a Notification Service Extension target to the Xcode project that downloads `aps.attachment-url` and attaches it to the notification content.** Wire-in points: `backend/routes/chat.js` (both `POST /chat/:matchId` paths) and `backend/server.js` `sendMissedCallPush`.
*   **Real-time story feed over Socket.io** (`backend/routes/stories.js`, `frontend/screens/ChatsScreen.tsx`): On story create/delete, the backend now emits `story:new` / `story:deleted` to every matched user's user-room (plus the owner's room) via `emitStoryEventToMatched(io, ownerId, event, payload)`. The active-feed cache for each matched user is invalidated *before* the emit, so when their client receives the event and refetches `/stories/active` it gets the fresh list. The Chats screen subscribes to both events; `story:new` reloads `fetchStories()` (and `fetchMyStories()` if the owner is us, e.g. posted from another device); `story:deleted` does the same so the tile disappears for everyone in real time. Brand-new matches still use the natural focus refresh (we only emit to currently-matched users at emit time).
*   **Tick progression: sent → delivered → seen** (`backend/server.js`, `backend/routes/chat.js`, `frontend/screens/ChatDetailScreen.tsx`): Bubble icons render `checkmark` (single ✓) for `sent`, `checkmark-done` (double ✓✓) for `delivered`, and `checkmark-done` colored `#4FC3F7` for `seen`. The semantic of "delivered" is now strictly **the message reached the recipient's device**, not "the server saved it". Two backend changes enforce that: (1) the legacy socket-send path (`server.js` `chat:message`) used to force `status: 'delivered'` immediately — now sets `'sent'`. (2) The HTTP send routes (`routes/chat.js` POST, both branches) only emit `chat:message-delivered` and persist `deliveredAt` when the recipient is in the `onlineUsers` map; otherwise the message stays in `sent` state. (3) On socket connect, `server.js` looks up the most recent 200 messages where `receiver === socket.userId`, `status === 'sent'`, `seen === false`, bulk-updates them to `delivered`, and emits `chat:message-delivered` to each unique sender's user-room — so the moment a recipient's device reconnects, all pending senders see their tick flip to ✓✓.
*   **Verification approval becomes visible in real time** (`backend/routes/verification.js`, `backend/routes/admin.js`, `frontend/screens/DiscoveryScreen.tsx`): Both approval handlers (admin dashboard at `/admin/verifications/:userId/approve` and self-service at `/verification/:userId/approve`) now `redis.delPattern('discovery:*')` after saving, wiping every discoverer's cached `/users/nearby` page so the newly-approved user appears within seconds instead of waiting up to 2 minutes for the per-key 120s TTL. The `user:verified` socket event was already being emitted to the approved user; the Discovery screen now subscribes to it and (a) for the **owner of the event** flips the local face-gate to `approved`, calls `fetchUser()` to update the AuthContext (so `isFaceVerified` and `verified` flip on), and silently reloads the deck — no app restart required; (b) for **any** verified event (someone else got approved) it also silently reloads the deck so they pop into the discovery stack. This fixes the "users show on Radar but not on Discovery" symptom where a freshly verified user wasn't being included in cached results.
*   **Stories cache invalidation across matched users** (`backend/routes/stories.js`): The `/stories/active` endpoint caches per-user with a 30s TTL (`STORY_ACTIVE_TTL`). Previously, posting a new story only cleared `stories:active:{ownerId}` — so any matched user that had recently fetched stories would keep seeing their stale (often empty) cached feed for up to 30s and the new story would silently fail to appear. Fix: `invalidateMatchedActiveStoryCaches(ownerUserId)` helper looks up every active match for the owner, builds the matched-user ID list, and `redis.del`s `stories:active:{matchedId}` for each one (plus the owner's own key). Wired into both `POST /stories` and `DELETE /stories/:storyId`. Effect: a story posted by Alice instantly invalidates Bob's cache (Bob is matched with Alice), so the very next time Bob opens the chats screen the story shows up. Note: brand-new matches (Bob just matched Alice for the first time) still inherit the natural 30s window — acceptable because the next refresh will be re-keyed against the current match list.
*   **Support center bubble alignment** (`frontend/screens/SupportMessagesScreen.tsx`, `backend/routes/support.js`): The chat thread renders messages on the LEFT for staff and RIGHT for the current user. The previous logic was `isStaff = msg.role === 'admin' || 'agent' || !!msg.adminName`. Older tickets in the database had user messages with `adminName` populated (legacy code path), which made user replies render on the LEFT and look like staff replies. Fix is two-part: (1) the chat renderer now uses `senderId === user._id` as the **primary** "is mine" check (immune to any stale `adminName`); falls back to `role === 'user'` only when senderId is missing (e.g. very old tickets). The optimistic message also stamps `senderId: user._id || user.id` so even the just-sent bubble renders correctly before reconciliation. (2) The `GET /api/support/my-messages` endpoint no longer copies `senderName` into `adminName` for user messages — it only sets `adminName` when the message is actually from a staff role (`admin` or `agent`). This stops future client surfaces from re-introducing the same bug.
*   **PremiumScreen conversion design** (`frontend/screens/PremiumScreen.tsx`): Conversion-focused layout — benefit-driven hero (eyebrow `AFROCONNECT PREMIUM`, headline `Stand Out. Get Noticed. Match Faster.`, subtitle pitching the diaspora positioning) backed by a multi-color animated radial glow and a layered crown (gold→orange→coral gradient with a pulsing yellow glow halo). A 3-column stat strip directly under the hero anchors social proof (`10× more matches`, `3× profile views`, `∞ daily likes`). The "Most Popular" tier (Monthly) is rendered inside a `popularWrapper` that adds (1) a pulsing coral outer glow and (2) a 3-color gradient border (`#FF6B6B → #FFA500 → #FFD700`) around a darker inner card so it visually dominates the other plans; the gradient saturation increases when that tier is selected. The CTA was upgraded to a multi-stop coral→orange gradient button with its own pulsing glow ring, an icon-tagged "{plan} · Save X%" pill above it (only when a savings tier is selected), and copy that flips between `Start Premium · $X/interval` and `You're Premium`. A `trustRow` (Secure payment / Cancel anytime / Private & safe) sits below the features grid for risk reduction. Section labels were converted from generic ("Choose Your Plan", "Premium Features") to benefit-led ("Pick the plan that fits you", "Everything you unlock") with secondary-text subheads. All IAP, restore, receipt-validation, and pricing logic is unchanged.
*   **"Seen at HH:MM" caption (Premium-gated)**: Below the bubble footer of the **most recent own message** (identified by `lastSeenOwnMessageId` in `frontend/screens/ChatDetailScreen.tsx`), an italic caption renders. For free users the caption shows just `Seen`; for Premium users (`user.premium.isActive === true`) it shows `Seen at HH:MM` using `formatTime(item.seenAt)`. The premium check is `const isPremium = !!user?.premium?.isActive` near the top of the component, included in `renderMessage`'s dep array so the caption re-renders on subscription state change. Only the timestamp portion is gated — the colored ticks remain available to all users. Backend feature flag: `seenAtTimestamp: true` in `iapWebhookService.PREMIUM_FEATURES` / `false` in `FREE_FEATURES`. Listed in the frontend `PREMIUM_FEATURES` array on `PremiumScreen.tsx` as "Seen-At Timestamps" with `clock-check-outline` icon.

## Voice & Video Call Behavior
*   **Callee-side missed-call notification** (`frontend/components/calls/IncomingCallHandler.tsx`): When the callee never picks up and the in-app `AUTO_DISMISS_MS` (30s) timer fires, the handler now does three things in sequence: (1) emits `socket call:missed` to the backend (existing behavior — fires the missed-call push to the **caller** + writes the chat-system message), (2) calls `dismissCallNotificationsForCaller(callerId)` to clear the still-presented "Incoming call…" lock-screen banner from this device, and (3) calls `presentLocalMissedCallNotification(data)` which schedules a local `expo-notifications` entry titled `Missed {voice|video} call` body `from {callerName}` with `data.type: 'missed_call'`. Without (2)+(3) the callee's lock-screen ringing notification would silently disappear and they'd have no surface-level indication they missed a call (the backend's missed-call push only goes to the caller in this flow, not back to the callee). Both helpers are component-level callbacks at the top of `IncomingCallHandler.tsx` and are also wired into the `showCallUI` `useCallback` dep array. The CallKeep `onEnd` (deliberate decline) path does NOT trigger the local missed-call notification — only the timeout path does.
*   **Voice call audio routing** (`frontend/screens/VoiceCallScreen.tsx`, `frontend/components/calls/IncomingCallHandler.tsx`): The WebView-hosted Agora voice client (`backend/public/agora-call.html`) cannot capture the mic when Android is in `MODE_IN_CALL` with `playThroughEarpieceAndroid: true` — that mode hands the mic to the system telephony stack and the WebRTC `getUserMedia` returns silent audio for both peers. The fix is two-part: (1) when ringtone playback stops in both the active screen and the incoming-call modal, we set `Audio.setAudioModeAsync({ playThroughEarpieceAndroid: false, allowsRecordingIOS: true })` so the mic stays with the WebView, and (2) the default `isSpeakerOn` state is now `true` so users hear the other party on the loudspeaker by default (the manual toggle still works). Without these two flips both sides could see "connected" but heard silence.
*   **Video call self-view PiP** (`frontend/screens/VideoCallScreen.tsx`, ~line 880): A previous implementation key-swapped a single `RtcSurfaceView` between the fullscreen and PiP slots when the remote joined. SurfaceView locks its z-order at attach time, so the swap left the local view buried under the remote view and the callee saw a black PiP. The current code mounts **two stable `RtcSurfaceView` instances** — one fullscreen (visible only while no remote is connected) and one PiP overlay with `zOrderMediaOverlay` set at mount time (visible only after remote joins). React never re-keys them; only their parent `View`'s `display` toggles. This guarantees the PiP always renders above the remote feed.
*   **Cold-start "screen disappears" guard** (`frontend/screens/VideoCallScreen.tsx` ~line 739, `frontend/screens/VoiceCallScreen.tsx` ~line 582): When the callee accepts via the OS-level VoIP push (CallKit / ConnectionService) and the app cold-starts straight into the call screen, the caller's `pendingCall` may have already timed out on the backend (35-second TTL in `backend/server.js`), causing it to broadcast `call:ended` to the room. Without protection that event would dismiss the screen the user just opened. Both call screens now arm an 8-second grace window in `socketService.onCallEnded` whenever `callAccepted === true` at mount time — `call:ended` events received during that window are ignored unless the remote/agora has already joined. This lets the Agora handshake complete before any stale lifecycle event can tear the UI down.
*   **Push-driven ringing while app is killed** (`backend/utils/fcmPush.js` `sendCallDataMessage`, `frontend/services/firebaseMessaging.ts` `setBackgroundMessageHandler`, `frontend/services/notifications.ts` `calls` channel): Incoming calls fan out across three channels in parallel — (a) APNs VoIP push for iOS (CallKit native UI via `react-native-voip-push-notification`), (b) FCM data-only message with `priority: high`, `ttl: 30s`, `collapseKey` for Android (the background handler invokes `RNCallKeep.displayIncomingCall` to show the native ConnectionService screen even when the app is killed), and (c) a regular Expo push on the high-importance `calls` channel (`AndroidImportance.MAX`, `sound: 'default'`, `bypassDnd: true`, vibration pattern) as a fallback ring/banner. Required env on Render: `FIREBASE_SERVICE_ACCOUNT` (full service-account JSON) and the configured Expo FCM server key. The `calls` Android notification channel is registered on first launch in `setupAndroidChannels`.
*   **CallKeep `selfManaged: true`** (`frontend/services/callkeep.ts`): The Android CallKeep config uses `selfManaged: true` (the WhatsApp/Telegram/Signal pattern for VoIP apps). With the previous `selfManaged: false`, ConnectionService routed every incoming call through Android's system telecom stack, which silently required the user to first toggle on a "Phone account" entry under Settings → Calling Accounts → AfroConnect. Almost no user does that, so the killed-app FCM wake fired but the full-screen call UI never appeared — only the fallback heads-up notification. With `selfManaged: true`, AfroConnect manages its own incoming-call surface via a high-priority full-screen-intent notification, no manual phone-account opt-in required. Both `initCallKeep` and `displayIncomingCall` are now `async` and the FCM background handler (in `firebaseMessaging.ts`) and `BackgroundCallTask` headless task (in `index.js`) `await` setup before calling `displayIncomingCall`, so the native bridge is fully ready before the call surface is shown. Android 14+ also requires `USE_FULL_SCREEN_INTENT` (declared in `app.json`); on devices where Google has revoked it, users may need to grant it manually under App info → Display over other apps. Battery-optimization whitelisting is recommended on aggressive OEMs (Xiaomi/Huawei/OnePlus).

## Local Dev Tooling
*   **Expo SDK pin**: The frontend is on **Expo SDK 55** (`expo@~55.0.17`, `@expo/metro-config@55.0.17`, `@expo/cli@55.0.26`). All `expo-*` companion packages are pinned to `~55.0.x`. SDK 55 is the version compatible with React 19.2 + React Native 0.83 + Metro 0.83.x — the stack this project uses. Earlier in the project history `expo` was pinned to `^52.0.0` while the rest of the SDK was already on 55, which caused `@expo/metro-config@0.19.x` to fail with `ERR_PACKAGE_PATH_NOT_EXPORTED` against newer metro internals (`metro-cache/src/stores/FileStore`, `metro/src/DeltaBundler/Serializers/sourceMapString`, etc.). Aligning the `expo` package itself to SDK 55 is the proper fix; do not downgrade.
*   **Patch script** (`frontend/scripts/patch-metro.js`): Postinstall step that opens up the `exports` field of `metro/`, `metro-cache/`, and `metro-transform-worker/` package.json files so older Expo CLI builds can still resolve their internal subpaths. With Expo 55 these patches are largely unused but kept as a safety net for any transitive consumer that still expects the old subpath access.
*   **Expo Go vs dev build**: `npm run start`, `npm run tunnel`, `npm run android`, `npm run ios` all pass `--go` so the QR is an `exp://...` URL that opens directly in **Expo Go**. To use a custom dev build (required for `react-native-agora`, `react-native-callkeep`, `react-native-iap`, `react-native-voip-push-notification` — none of these work in Expo Go), use `npm run tunnel:dev` instead.
*   **Backend URL config**: The frontend reads its API base URL from `process.env.EXPO_PUBLIC_API_URL` in `constants/config.ts`. Expo only inlines vars that start with `EXPO_PUBLIC_` and live in `frontend/.env` (NOT the repo root, NOT the backend folder). To point the app at the Render backend, create `frontend/.env` containing:
    ```
    EXPO_PUBLIC_API_URL=https://<your-render-app>.onrender.com
    ```
    Then fully restart Metro (`Ctrl+C` then `npm run tunnel` again). Without this, login fails with "Couldn't connect to server" because there is no fallback URL — the config throws a clear error in dev logs.
*   **Expo start wrapper** (`frontend/scripts/expo-start.js`): All `npm` scripts (`start`, `tunnel`, `android`, `ios`, `web`, `dev`) go through this Node wrapper instead of calling `npx expo start` directly. It does two portable things:
    1. Sets `EXPO_FORCE_WEBCONTAINER_ENV=1` so `--tunnel` uses Expo's modern WebSocket tunnel (`*.boltexpo.dev`) instead of the deprecated ngrok 2.3.41 client bundled with Expo. The legacy ngrok v2 protocol no longer reaches ngrok's servers, so `instance.connect()` rejects with no `error.response`, which previously surfaced as `CommandError: TypeError: Cannot read properties of undefined (reading 'body')` from inside `@expo/ngrok/src/client.js`.
    2. If a `replit.nix` is present (Replit / Nix sandbox), it shells out to `nix eval` and prepends every dep's `lib` directory to `LD_LIBRARY_PATH` so the `@react-native/debugger-shell` dotslash binary can find `libglib-2.0.so.0`, `libnspr4.so`, `libgit2-*.so`, etc. On a normal Linux/macOS dev machine the block is a no-op.
*   **Ngrok client patch** (`frontend/scripts/patch-expo-ngrok.js`): Postinstall step that hardens `@expo/ngrok/src/client.js` so when `error.response` is undefined it throws a clear "Re-run with the modern WebSocket tunnel" message instead of the opaque TypeError. Idempotent — guarded with an `AFRO_NGROK_PATCHED` marker. Chained from `package.json` `postinstall` after `patch-metro.js`.
*   **Local Linux requirement**: If running on a fresh Linux dev machine and the React Native DevTools shell complains about missing `libglib-2.0.so.0` or similar, install the Chromium runtime libs via your distro (`apt install libglib2.0-0 libnss3 libnspr4 libatk1.0-0 libatk-bridge2.0-0 libcups2 libdrm2 libxkbcommon0 libxcomposite1 libxdamage1 libxfixes3 libxrandr2 libgbm1 libpango-1.0-0 libcairo2 libasound2`). Not needed on macOS / Windows.

## Chat Message Ordering (ChatDetailScreen)
*   **Chronological FlashList (no `inverted`)** — FlashList v2 (`2.0.2`) has known issues with the `inverted` prop where the visual order of items can flip incorrectly under certain layout passes, surfacing as "newest message at the top, oldest at the bottom" — the opposite of every standard messaging app. We render the message list in plain chronological order (oldest first, newest last) and rely on FlashList v2's `maintainVisibleContentPosition={{ autoscrollToBottomThreshold: 0.2, startRenderingFromBottom: true }}` to anchor the initial render at the bottom and auto-follow new incoming messages when the user is near the latest one.
*   **Scroll behaviour**: `scrollToBottom()` uses `scrollToEnd({ animated })` (newest message is the last item). The "near bottom" detection in `onScroll` uses `contentSize.height - (contentOffset.y + layoutMeasurement.height) < 120` — distance from the bottom edge — instead of the previous `y < 120` which only worked while `inverted` was on. Loading older history is triggered by detecting `contentOffset.y < 120` (near the top of the list) inside `onScroll` plus a "Load older messages" button rendered as `ListHeaderComponent`. `onEndReached` was removed because it now corresponds to the bottom (newest), not the top (oldest).
*   **`scrollToMessage(messageId)`** uses the original chronological index from `enrichedMessages` (no inversion math). Reply-jump and search-jump both work directly with this index.

## Discovery / Radar Parity
*   The discovery deck (`/users/nearby` + `frontend/screens/DiscoveryScreen.tsx`) used to apply auto-filters that the radar feed (`/radar/nearby-users` + `LoveRadarScreen`) did not, which manifested as "users show on radar but never appear in discovery":
    *   **Auto-applied lifestyle filters** (REMOVED): the frontend was sending the user's own `lookingFor`, `religion`, `smoking`, `drinking`, and `wantsKids` as hard query filters on every discovery request. This excluded anyone who hadn't filled in the matching field (or whose value differed). These auto-sends are removed — the backend still honours them when explicitly passed in via the discovery filter modal, but they are never applied silently. Gender preference and age range are still auto-applied because they are core matching axes, not lifestyle preferences.
*   **Distance cap** (kept in sync between both endpoints): free users are capped at `50km` on both discovery (`FREE_MAX_DISTANCE_KM` in `backend/routes/users.js`) and radar (`radarMaxKm` in `backend/routes/radar.js`). Premium users can scan up to their preferred max (capped at `100km` on radar, unbounded on discovery via passport / saved location / Global). The 50km cap is enforced server-side on both endpoints, with the client also clamping the requested distance for cache-key stability.
*   The exclusion sets (blocked, blocked-by, swipedRight/Left, active matches) are already shared between the two endpoints. Pending FriendRequests are excluded from radar but NOT from discovery, so a swipe on a pending-incoming-like will instantly mutual-match instead of being hidden.
## Read Receipts (✓ → ✓✓ → ✓✓ blue)
*   **Backend `chat:message-delivered` emit destination** (`backend/routes/chat.js`): The "delivered" socket event is emitted to the **sender's** user-room (and the matchId room as a fallback), NOT the receiver's room. The sender is the one whose UI bubble needs to update from one tick (sent) to two ticks (delivered) — emitting it to the receiver had no effect on UI and left the sender's bubble stuck on a single tick forever. This applies to both `POST /:matchId` send endpoints in `chat.js`.
*   **Frontend handler**: `ChatDetailScreen.tsx` listens for `chat:message-delivered` (status → 'delivered') and `chat:message-read` (status → 'seen', renders blue checkmarks). The receiver fires `PUT /chat/:matchId/read` on chat open AND when receiving a new message while the chat is open, which updates `seen=true, status='seen'` on the message and broadcasts `chat:message-read` to both the matchId room and the sender's user-room.

## Stories Visibility
*   `GET /stories/active` (`backend/routes/stories.js`): Stories are visible to ALL active matches, not just matches you've already exchanged messages with. The previous `chattedUserIds` filter was a chicken-and-egg restriction that hid your story from a fresh match until someone broke the ice.

## Voice Call Codec (Agora WebView Bridge)
*   `backend/public/agora-call.html`: `AgoraRTC.createClient` uses `codec: 'vp8'` (the recommended default for Communication mode). Previously used `'h264'` which caused some Android WebViews to fail SDP negotiation, resulting in both sides going silent after the receiver accepted a voice call.

## Incoming Call UI — Foreground vs Background
*   `frontend/components/calls/IncomingCallHandler.tsx`: The native CallKit / ConnectionService full-screen takeover (`displayIncomingCall`) is only triggered when the app is **NOT in foreground** (checked via `AppState.currentState`). When the app is in foreground, only the in-app banner Modal slides down — no native call screen. When the app is backgrounded or killed, the system call UI still appears via VoIP push (iOS) or FCM data message → Firebase background handler (Android), so the user is still notified.
